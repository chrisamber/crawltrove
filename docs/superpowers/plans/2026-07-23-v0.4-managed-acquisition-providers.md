# CrawlTrove v0.4 Managed Acquisition Providers Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add dedicated Firecrawl, Bright Data, and Browserbase acquisition paths behind deterministic capability routing, native-unit budgets, durable attempt auditing, and the existing local extraction pipeline.

**Architecture:** A small provider protocol normalizes transport outcomes but does not hide native usage meters. The router starts with local/owned capability routes, atomically reserves the selected provider's worst-case native usage, calls one adapter, revalidates the returned target, and reconciles the append-only attempt. Managed HTML is always processed by `WebScraper._build_result()` so CrawlTrove remains the extraction and persistence system of record.

**Tech Stack:** Python 3.11, httpx, Playwright CDP, FastAPI/Pydantic, asyncpg/PostgreSQL 16, pytest `MockTransport`; no vendor SDK dependency.

## Global Constraints

- Plans 1–3 are merged; use their strict crawl repository, worker, attempts ledger, encrypted profiles, and provider-neutral `SessionBackend`.
- Managed acquisition is available to durable `/api/crawl` only in v0.4. `POST /api/scrape` remains local and database-independent.
- Dedicated adapters are Firecrawl Scrape/Interact, Bright Data Web Unlocker, and Browserbase Sessions.
- Missing credentials disable only that adapter. Health/admin output reports disabled state, `auto` skips it, and explicit selection returns stable HTTP 503 `provider_unavailable` before work or usage reservation.
- Public `acquisition` input exposes provider/auto, native budgets, max attempts, session profile, human permission, and existing browser/location fields only; there is no arbitrary provider-option passthrough.
- Exact route keys are `local_http`, `owned_proxy_http`, `local_browser`, `brightdata_unlocker`, `firecrawl_scrape`, `browserbase_session`, and `firecrawl_interact` everywhere: capabilities, attempts, logs, metrics, tests, and dashboard.
- `auto` maps ordinary pages to local HTTP; static access blocks to owned proxy then Bright Data; rendering needs to owned browser then Firecrawl; persistent interaction/human work to Browserbase then Firecrawl Interact.
- Only classified retryable failures advance the router. Policy/input/permanent failures stop immediately.
- Default is four total attempts per URL and at most two per route; callers may lower but not raise either cap.
- Core validates the initial URL before a paid call and revalidates every provider-reported final URL and returned links. Do not claim DNS-pinning guarantees inside a managed provider.
- Provider calls use hard pre-call reservations. Therefore v0.4 uses only fixed-meter Firecrawl features, one-request Bright Data calls, and Browserbase direct egress with a hard session timeout.
- Browserbase's API does not expose a hard per-session proxy-byte cap. To preserve the hard-budget contract, v0.4 MUST NOT enable Browserbase-managed proxies; `proxyBytes` is accepted only as zero. Bright Data/owned egress cover proxy acquisition.
- Firecrawl uses basic scrape (one credit/page) and code-only Interact (two credits/browser minute, bounded by timeout). Do not enable enhanced/JSON/AI-prompt modes with variable credits.
- Bright Data reserves one request before each Web Unlocker call and reconciles one request whether the zone bills only success or is configured to bill all requests.
- Provider live URLs, API keys, Browserbase connection URLs, Firecrawl interactive URLs, cookies, and billing/debug headers are ephemeral secrets and never persisted or logged.

```python
@dataclass(frozen=True)
class NativeCost:
    values: Mapping[str, int | float]
    estimated: bool = False


@dataclass(frozen=True)
class ProviderRequest:
    url: str
    route: str
    timeout_seconds: int
    only_main_content: bool
    session_profile: str | None = None


@dataclass(frozen=True)
class ProviderResult:
    raw_html: str
    final_url: str
    status_code: int | None
    native_cost: NativeCost
    remote_session_id: str | None = None


@dataclass(frozen=True)
class ProviderFailure(Exception):
    code: str
    retryable: bool
    native_cost: NativeCost
    status_code: int | None = None


class ProviderAdapter(Protocol):
    name: str
    routes: frozenset[str]
    def available(self) -> bool:
        raise NotImplementedError
    def reserve_cost(self, request: ProviderRequest) -> NativeCost:
        raise NotImplementedError
    async def acquire(self, request: ProviderRequest) -> ProviderResult:
        raise NotImplementedError
    async def cancel(self, remote_id: str) -> None:
        raise NotImplementedError
```

```sql
CREATE TABLE crawl_provider_usage (
    job_id UUID NOT NULL REFERENCES crawl_jobs(id) ON DELETE CASCADE,
    provider TEXT NOT NULL,
    meter TEXT NOT NULL,
    limit_value NUMERIC NOT NULL CHECK (limit_value >= 0),
    reserved_value NUMERIC NOT NULL DEFAULT 0 CHECK (reserved_value >= 0),
    consumed_value NUMERIC NOT NULL DEFAULT 0 CHECK (consumed_value >= 0),
    PRIMARY KEY (job_id, provider, meter),
    CHECK (reserved_value + consumed_value <= limit_value)
);

ALTER TABLE acquisition_attempts
    ADD COLUMN remote_id TEXT,
    ADD COLUMN lease_token UUID,
    ADD COLUMN cost_estimated BOOLEAN NOT NULL DEFAULT false;
```

- [ ] **Step 4: Implement fenced reserve/reconcile transactions**

`reserve_acquisition_attempt()` locks task/job/usage rows, verifies the current lease, counts all attempts against four and the exact route against two, checks every meter has enough `limit - consumed - reserved`, increments reservations, and inserts the append-only attempt with its lease token. `finish_acquisition_attempt()` locks the attempt and usage rows, rejects a stale lease, requires actual keys to equal reserved keys and actual values not exceed reservations, converts reservation to consumption, and finalizes the attempt once. Unknown meters or overspend become `provider_protocol_error` and disable that adapter until health recovery.

## Plan 4 Exit Gate

- `auto` follows the agreed exact capability map and never spends provider usage on a permanent/policy failure.
- Explicit unavailable providers fail before job creation; optional missing providers do not prevent startup.
- Every provider call has one append-only attempt, one hard pre-call native reservation, and one reconciliation.
- Firecrawl uses only fixed-meter basic/code features; Bright Data meters each call; Browserbase uses hard-timeout direct egress with managed proxies disabled.
- All managed content passes through URL revalidation and CrawlTrove's local extraction/signals pipeline.
- Provider cancellation is attempted promptly and stale completions cannot commit.
- Hermetic tests are mandatory in public CI; one real, budget-capped smoke for each provider blocks the private v0.4 release.
