# CrawlTrove v0.4 Dashboard, Evaluation, and Release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the complete v0.4 acquisition system observable and operable, automate the CrawlTrove/Firecrawl/Crawl4AI comparison, and ship v0.4 only after durability, security, provider, and deployment gates pass.

**Architecture:** Low-cardinality metrics and authenticated admin APIs expose the durable state already held in PostgreSQL; the existing dashboard adds acquisition detail and intervention controls without becoming a secret manager. A standalone evaluation package runs direct adapters against checked-in cases and writes ignored machine-readable reports. Release workflows keep hermetic checks public and run paid provider smoke tests only in a protected private environment.

**Tech Stack:** FastAPI, asyncpg/PostgreSQL 16, prometheus-client, Next.js 16/React 19/TypeScript 5.9, Python 3.11 evaluation CLI, pytest, Node test runner, GitHub Actions, Docker Compose.

## Global Constraints

- Plans 1–4 are merged and individually green before this plan begins.
- The dashboard shows job/task state, attempts, block reasons, native usage, workers, and live intervention; provider credentials and worker enrollment remain CLI/env only.
- Keep the current top-level dashboard navigation. Acquisition detail belongs in existing Crawl and Run inspectors.
- Treat `completed`, `partial`, `failed`, `cancelled`, and `timed_out` as terminal. `waiting_input` is active and actionable.
- Public page cap remains 100 in API and dashboard.
- Metrics use stable low-cardinality labels only; URLs, origins, job IDs, task IDs, worker IDs, session IDs, and exception class names are forbidden labels.
- Structured logs may contain IDs for correlation but never raw bodies, authorization, cookies, API keys, session URLs, full query strings, or provider billing/debug headers.
- Default retention deletes only abandoned temporary uploads after 24 hours. Completed jobs, attempts, and referenced artifacts remain until explicit/operator-configured purge; diagnostic raw bodies remain off.
- The acquisition benchmark compares CrawlTrove, Firecrawl, and Crawl4AI on the existing simple HTML, JavaScript, plain-text, and PDF classes, five cache-disabled runs each.
- Benchmark quality/correctness is informational and not a release gate because public network/provider conditions vary.
- One budget-capped live smoke each for Firecrawl, Bright Data, and Browserbase is a private v0.4 release gate.
- Builds and test suites sharing caches, PostgreSQL, Compose, or browser state run sequentially.
- `shipped` means v0.4 is tagged and publicly distributed/reachable, not merely committed or passing locally.
- No generated benchmark reports from `tmp/`, provider responses, credentials, or local data are committed.
